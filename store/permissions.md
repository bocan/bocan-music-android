# Permissions rationale

Every permission the app declares, why it is there, and what to tell the Play reviewer.
The list is deliberately minimal: no storage permissions, no location, no advertising ID,
no `MANAGE_EXTERNAL_STORAGE`. Synced media lives in app-specific external storage, which
needs no permission at all.

## Declared permissions

| Permission | Why the app needs it |
|------------|----------------------|
| `INTERNET` | The sync client connects to the paired Mac over TLS on the local network. |
| `ACCESS_NETWORK_STATE` | Detect connectivity so sync and scrobble queues drain when the network returns, and back off when it is gone. |
| `ACCESS_WIFI_STATE` | Confirm the phone is on Wi-Fi and identify the local network for discovery of the paired Mac. |
| `CHANGE_WIFI_MULTICAST_STATE` | Hold a multicast lock so mDNS/NSD discovery of the Mac works; released as soon as discovery ends. |
| `FOREGROUND_SERVICE` | Base permission for the playback and sync foreground services. |
| `FOREGROUND_SERVICE_MEDIA_PLAYBACK` | Typed permission for the media playback foreground service (audio continues with the screen off). |
| `FOREGROUND_SERVICE_DATA_SYNC` | Typed permission for the sync foreground service (a library sync survives the app going to the background). |
| `POST_NOTIFICATIONS` | The media notification with transport controls, and the sync-progress notification. Requested at runtime on Android 13+. |
| `WAKE_LOCK` | Keep the CPU awake during active playback and during an in-progress sync only; released when both are idle (zero wakelocks while paused, per the performance baselines). |
| `RECEIVE_BOOT_COMPLETED` | Reschedule background sync work with WorkManager after a reboot. |
| `REORDER_TASKS` | Bring the existing player task to the front from the media notification and Android Auto rather than stacking a new one. |

The `*.DYNAMIC_RECEIVER_NOT_EXPORTED_PERMISSION` entries are generated by AndroidX for
runtime-registered `RECEIVER_NOT_EXPORTED` receivers (and the `.test` variants only exist in
test manifests). They are not user-facing and need no rationale.

## Foreground service declarations (Play Console)

Both foreground service types must be declared in the Play Console with a short demo, and
both are genuinely used:

- **`mediaPlayback`**: the Media3 `MediaLibraryService` that plays the user's library with
  the screen off and drives the media notification, lock screen, Bluetooth, and Android
  Auto. Demo video: open the library, tap a track, lock the phone, show audio continuing
  with the notification controls. Record to `store/`.
- **`dataSync`**: the sync service that pulls new and changed media from the paired Mac.
  Demo video: Settings, Sync Now, show the sync-progress notification advancing while the
  app is backgrounded. Record to `store/`.

Neither service runs while idle: playback stops its service when playback stops, and the
sync service stops when the sync completes, so no foreground service is held at rest.

## What is deliberately absent

- No `READ_MEDIA_AUDIO` / `READ_EXTERNAL_STORAGE` / `MANAGE_EXTERNAL_STORAGE`: the app only
  ever reads its own app-specific external files directory, which requires no permission.
- No location, contacts, camera, microphone, or Bluetooth-connect permissions: media routing
  to Bluetooth is handled by the platform media session without app-held Bluetooth access.
- No advertising or device-ID permission: there is no analytics or ads code to serve.
